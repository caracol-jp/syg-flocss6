# commit-message

直近の変更を整理し、コミットメッセージ案を提示するスキル。**ユーザー確認なしのコミットは禁止**。

## 使い方

ユーザーから「コミットメッセージを考えて」「コミット内容まとめて」等の指示を受けたら、以下のステップで進める。

### 1. 変更内容の把握

並列で以下を実行：

```bash
git status
git diff --stat
git diff
git log -10 --oneline
```

- ステージング済み / 未ステージング両方の差分を見る
- 直近のコミットメッセージのスタイル（言語・形式・接頭辞）を踏襲する

### 2. コミットメッセージ案の作成

#### タイトル（1 行目）

- **日本語**で記述（CLAUDE.md の共通ルール）
- 簡潔に変更の要点を表す（50 文字以内目安）
- 接頭辞があれば過去ログのスタイルを継承（例: `feat:`、`fix:`、`docs:`、`refactor:`）。スタイルが無ければ接頭辞なしで OK
- 「なぜ」よりも「何が変わったか」を端的に

#### 本文（必要なら）

- 1 行空けて、変更理由・背景・影響範囲を箇条書きで補足
- 1 行 72 文字程度で折り返し
- 変更ファイル数が多い場合や、設計判断を含む場合に追加

### 3. ユーザー確認（必須）

タイトル案・本文案を提示し、**必ずユーザーに OK / 修正指示を求める**。

```
以下の内容でコミットしてよいですか？

[タイトル案]

[本文案（あれば）]

確認: そのまま実行 / 修正 / キャンセル
```

ユーザーが「OK」「コミットして」等の明確な承認を返した場合のみ次に進む。
曖昧な返答（「うん」「はい」だけなど）でも、文脈で承認と判断できる場合は進めて良い。**確認をスキップして自動実行することは絶対にしない**。

### 4. コミット実行

確認が取れたら、HEREDOC でメッセージを渡してコミットする：

```bash
git commit -m "$(cat <<'EOF'
[タイトル]

[本文]
EOF
)"
```

### 5. バージョンタグ付け（`package.json` の version を変更した場合）

今回のコミットで `package.json` の `version` を更新した場合は、その番号でタグを付ける。

- タグ名・メッセージともに **`v` 接頭辞は付けない**（既存タグの形式に合わせる。`git tag --sort=-creatordate` で確認。本リポジトリは `1.0.1` 形式）
- 注釈付きタグで付与する：

```bash
git tag -a 1.0.1 -m "1.0.1"
```

- タグは**コミット完了後**に付ける（対象は直前のコミット）
- `version` を変更していない場合はこのステップをスキップする

### 6. 完了報告

`git status` を実行してコミットが反映されていることを確認し、ユーザーにコミットハッシュを伝える。タグを付けた場合はタグ名も伝える。

## ルール

- **必ずユーザー確認を取る**。Claude Code のグローバル指示と一致させ、確認なしの自動コミットは禁止
- **`git push` は絶対に行わない**（明示的な指示があった場合のみ別途検討）
- **`--amend` は使わない**（過去のコミットを書き換える危険があるため、新規コミットを推奨）
- **`--no-verify` は使わない**（pre-commit hook を尊重）
- **ステージングは `git add <ファイル名>` で個別に**。`git add -A` や `git add .` は機密ファイルや想定外のファイルを混ぜる可能性があるため避ける
- **`.env` 等の機密ファイルが含まれていないか確認**してからコミットする
- **言語は日本語**（CLAUDE.md 共通ルール）

## 参考

- 既存のコミット履歴: `git log --oneline -20` で確認
- Claude Code が pre-commit hook を実行して失敗した場合は、修正後に **新しいコミット**を作成する（`--amend` は使わない）
